# markdown2web 测试规格（Test Spec）

- 日期：2026-04-08
- 对应方案：`prd-markdown2web-2026-04-08.md`

## 1. 测试目标

验证该系统是否能稳定完成以下闭环：

1. Markdown 内容进入系统
2. 构建为网站页面
3. 链接、图片、附件可正常工作
4. 文件变更能自动触发更新
5. 后台管理可控且安全

## 2. 测试分层

## 2.1 单元测试

覆盖模块：

- `src/content/front_matter.rs`
- `src/content/markdown.rs`
- `src/content/links.rs`
- `src/content/assets.rs`
- `src/build/cache.rs`

用例：

1. 正常解析 Front Matter。
2. 缺失 Front Matter 时使用默认值。
3. 从标题自动生成 slug。
4. 自定义 slug 优先于自动 slug。
5. 标准 Markdown 链接被正确重写为 `/notes/{slug}`。
6. `[[Wiki Link]]` 被解析为正确目标。
7. 缺失目标笔记时产生断链告警。
8. 图片路径被正确解析并映射到站点资源目录。
9. 附件路径被正确映射到公开 URL。
10. 同一资源重复构建时不会被重复复制。
11. 构建缓存命中时跳过无变化页面。
12. 文件 hash 变化时正确触发重建。

## 2.2 集成测试

覆盖模块：

- `src/web/public.rs`
- `src/web/admin.rs`
- `src/store/sqlite.rs`
- `src/build/pipeline.rs`
- `src/build/watcher.rs`

用例：

1. 启动服务后首页可访问。
2. 初始化内容目录后，笔记列表页正确列出 Markdown 文件。
3. 上传一个 Markdown 文件后，单篇页面可访问。
4. 上传包含图片的 Markdown 后，页面可显示图片。
5. 上传包含 PDF/附件的 Markdown 后，附件链接可下载。
6. 两篇互相链接的笔记上传后，可双向跳转。
7. 删除某篇笔记后，断链报告或 alias 逻辑生效。
8. 后台修改已存在笔记后，公共页显示更新内容。
9. 全量重建后，列表页、标签页、详情页一致更新。
10. 构建失败时，后台日志可见错误信息。

## 2.3 Watcher / 自动构建测试

目标：验证“笔记目录变化 -> 自动更新网页”。

用例：

1. 修改 `content/notes/a.md` 后，5 秒内对应页面内容更新。
2. 重命名文件后，目标页面 URL 与链接图正确更新。
3. 连续快速保存同一文件时，不产生构建风暴。
4. 修改被多篇文章引用的文件时，相关反向链接也同步更新。
5. 修改资源文件（图片）后，页面展示更新。
6. 外部新增 Markdown 文件后，自动出现在列表页。
7. 外部删除 Markdown 文件后，列表页和详情页同步移除。

## 2.4 端到端测试（E2E）

建议从用户视角验证完整闭环：

1. 管理员登录后台。
2. 上传一篇带 Front Matter、图片、附件、内部链接的 Markdown。
3. 打开公开页面，确认：
   - 标题正确
   - 正文正确
   - 目录正确
   - 图片正确
   - 附件正确
   - 内部链接正确
4. 回到后台更新内容。
5. 再次访问公开页，确认内容变化已生效。
6. 在磁盘中直接修改文件，确认 watcher 自动更新页面。

## 2.5 安全测试

用例：

1. 未登录无法访问后台管理页。
2. 密码错误无法登录。
3. 上传超出大小限制的文件被拒绝。
4. 上传非法类型文件被拒绝或隔离。
5. Markdown 中恶意脚本不会直接造成前台 XSS（按所选 HTML 策略验证）。
6. Session cookie 具备安全属性。

## 2.6 UI / 体验测试

用例：

1. 桌面端页面结构清晰、可读性良好。
2. 移动端排版不溢出、不遮挡。
3. 深浅色主题切换正常（若首版实现）。
4. 长文目录导航可用。
5. 搜索结果与标签导航可用（若首版实现）。

## 3. 非功能测试

## 3.1 性能

目标建议：

1. 单篇普通笔记上传后，增量更新在 5 秒内完成。
2. 100 篇中小体量笔记的全量构建在可接受范围内完成。
3. 页面首屏响应保持轻量，避免不必要 JS 负担。

## 3.2 稳定性

1. 连续多次修改同一文件不造成崩溃。
2. 构建失败不会影响已生成页面继续服务。
3. 服务重启后能重新扫描内容目录并恢复索引。

## 3.3 可恢复性

1. SQLite 与内容目录备份可恢复。
2. 删除缓存后可通过全量重建恢复站点。

## 4. 测试数据设计

建议准备以下样本：

1. 纯文本笔记
2. 带 Front Matter 的笔记
3. 带多层标题和目录的长文
4. 带 Wiki Link 的笔记
5. 带相对路径链接的笔记
6. 带图片的笔记
7. 带 PDF/ZIP 附件的笔记
8. 含中文文件名与中文标题的笔记
9. 含断链的异常笔记
10. 含大图片资源的笔记

## 5. 验收门槛

在进入“可用首版”前，至少满足：

1. 关键单元测试通过。
2. 至少 1 条上传链路集成测试通过。
3. 至少 1 条 watcher 自动更新测试通过。
4. 至少 1 条带图片和内部链接的 E2E 场景通过。
5. 后台认证与上传限制生效。

## 6. 建议测试执行顺序

1. 先锁定 `content/*` 与 `generated/*` 的目录契约。
2. 再测试 Markdown 解析与链接重写。
3. 再测试构建管线。
4. 再测试后台上传与编辑。
5. 最后测试 watcher 自动更新与完整 E2E 流程。

## 7. 首版最重要的回归测试

每次改动以下模块时，都应回归：

1. 上传 Markdown -> 页面可访问
2. 修改笔记文件 -> 页面自动更新
3. 内部链接 -> 正常跳转
4. 图片/附件 -> 正常展示
5. 删除/改名 -> 断链或 alias 行为正确
